# Звіт про продуктивність: GamerIS Pro (Лабораторна робота №8)

## 1. Інструменти та методологія
- **Профілювання CPU/Runtime:** Chrome DevTools (Performance Tab).
- **Аналіз рендерингу:** React Developer Tools (Profiler).
- **Метрики:** Scripting Time, Rendering Time, Total Task Duration.
- **Сценарій тестування:** Завантаження стрес-тестового набору даних (1000 об'єктів у каталозі) та інтенсивна взаємодія (10+ послідовних операцій додавання у кошик).

## 2. Базові метрики (Baseline - до оптимізації)
Результати профілювання при стрес-тесті без використання мемоїзації:
- **Scripting Time:** 5 741 ms (Критичне блокування основного потоку).
- **Rendering Time:** 1 961 ms.
- **Total Task Time:** 21 503 ms.
- **Відгук інтерфейсу:** Помітні затримки (Input Lag) при кліках на кнопки "Купити".

## 3. Виявлені "гарячі точки" (Hot Spots)
1. **Unnecessary List Re-renders:** Кожен доданий товар спричиняв повний перерендер усіх 1000 карток `ProductCard` через зміну стану `cart` у батьківському компоненті `App`.
2. **Expensive Total Calculation:** Функція `calculateTotal` виконувалася при кожному циклі рендерингу, навіть якщо вміст кошика не змінювався.
3. **Synchronous Logging Context:** Складна логіка формування логів у реальному часі створювала додаткове навантаження на Scripting Time.

## 4. Реалізовані оптимізації (Етап 100%)
Для усунення виявлених проблем було застосовано наступні підходи:
- **React.memo**: Компонент `ProductCard` обгорнуто у `memo`, що дозволило React пропускати рендеринг карток, пропси яких не змінилися. Складність оновлення впала з $O(N)$ до $O(1)$.
- **useMemo**: Розрахунок змінної `total` мемоїзовано. Тепер функція викликається лише при фактичній зміні масиву `cart`.
- **useCallback**: Функція-обробник `addToCart` стабілізована, щоб запобігти руйнуванню мемоїзації дочірніх компонентів при кожному рендері `App`.

## 5. Порівняльні результати (До та Після)

| Метрика            | До оптимізації (Baseline) | Після оптимізації (Optimized) | Покращення |
|--------------------|---------------------------|-------------------------------|------------|
| **Scripting (JS)** | 5 741 ms                  | 919 ms                        | **~84.0%** |
| **Rendering**      | 1 961 ms                  | 3 187 ms*                     | -          |
| **Total Task Time**| 21 503 ms                 | 18 634 ms                     | **~13.3%** |

> **Примітка до Rendering:** Збільшення часу рендерингу на графіку зумовлене тривалим малюванням (Painting) великої кількості DOM-вузлів при першому завантаженні, проте **Scripting Time**, що відповідає за швидкість відгуку на дії користувача, скоротився у 6 разів.

### Розрахунок відсотка покращення Scripting Time:
$$Improvement = \frac{T_{baseline} - T_{optimized}}{T_{baseline}} \times 100\%$$
$$Improvement = \frac{5741 - 919}{5741} \times 100 \approx 83.99\%$$

## 6. Аналіз нових "гарячих точок"
Після успішної оптимізації оновлень, новими вузькими місцями стали:
1. **Initial Hydration**: Первинне завантаження 1000 елементів у DOM все ще займає значний час (Rendering). Для подальшого покращення варто розглянути **Virtual Scrolling** (наприклад, `react-window`).
2. **Bundle Size**: Використання додаткових хуків та мемоїзації незначно збільшує обсяг коду, що можна вирішити через **Tree Shaking**.

## 7. Висновок
Впроваджені методи мемоїзації дозволили радикально (на 84%) покращити продуктивність JavaScript-частини застосунку. GamerIS Pro тепер демонструє стабільну роботу та миттєвий відгук на дії користувача навіть при роботі з великими масивами даних.
