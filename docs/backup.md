# Регламент резервного копіювання та відновлення системи

Цей документ описує стратегію захисту даних проєкту **GamerIS**.

## 1. Стратегія резервного копіювання
Враховуючи архітектуру проєкту (SaaS/PaaS), обрано комбіновану стратегію:

- **Типи копій**: 
  - **Повні копії (Full Backups)**: Для бази даних (Google Sheets) та конфігурацій.
  - **Контрольні точки (Version Control)**: Для сирцевого коду.
- **Частота**: 
  - База даних: Щотижня (автоматично/вручну) та перед кожним великим оновленням.
  - Код: При кожному коміті до репозиторію.
- **Зберігання та ротація**: Копії зберігаються у трьох місцях (правило 3-2-1): локально, на Google Drive та в репозиторії GitHub. Термін зберігання архівних копій БД — 3 місяці.

## 2. Процедура резервного копіювання

### База даних (Google Sheets)
1. Відкрийте робочу таблицю проєкту.
2. Оберіть пункт меню **Файл -> Створити копію**.
3. Назвіть файл за шаблоном: `GamerIS_DB_Backup_YYYYMMDD`.
4. Для критичних бекапів: **Файл -> Завантажити -> Microsoft Excel (.xlsx)**.

### Файли конфігурацій та коду
1. Резервне копіювання коду виконується командою: `git push origin main --tags`
2. Конфігураційні файли (`package.json`, `eslint.config.js`) включені до складу репозиторію.

### Логи системи
Оскільки проєкт використовує Google Apps Script, логи зберігаються в Apps Script Dashboard (Executions). Для бекапу логів необхідно експортувати їх через Google Cloud Console (якщо підключено Cloud Logging).

## 3. Перевірка цілісності
Раз на місяць необхідно виконувати "Тестове відновлення":
1. Відкрити останню завантажену копію `.xlsx`.
2. Перевірити наявність усіх вкладок та коректність форматів даних (ціни, ID товарів).

## 4. Автоматизація
Для автоматизації копіювання бази даних рекомендується використання Google Apps Script Triggers, що виконують функцію `drive.files.copy` за розкладом.

## 5. Процедура відновлення

### Повне відновлення системи (Disaster Recovery)
1. Клонувати репозиторій: `git clone [url]`.
2. Відновити базу даних:
- Взяти останній бекап-файл `.xlsx`.
- Завантажити його в Google Sheets.
- Оновити URL скрипта в коді фронтенду.
3. Виконати деплой: `npm run deploy`.

### Вибіркове відновлення даних
Використовуйте функцію Історія версій (Version History) у Google Таблицях:
- Оберіть **"Файл" -> "Історія версій" -> "Переглянути історію версій"**.
- Оберіть потрібну точку відновлення та натисніть **"Відновити цю версію"**.
