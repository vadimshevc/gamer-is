# Інструкція з оновлення та підтримки системи

Цей документ містить регламент оновлення проєкту **GamerIS** для DevOps-інженерів.

## 1. Підготовка до оновлення
Перш ніж вносити зміни у виробниче середовище (Production), необхідно виконати наступні дії:

1.  **Створення резервних копій**:
    - **Код**: Переконайтеся, що поточна стабільна версія зафіксована тегом у Git (наприклад, `v1.0.0`).
    - **Дані (Google Sheets)**: Створіть копію робочої таблиці («Файл» -> «Створити копію») та назвіть її `Backup_YYYY_MM_DD`. Це необхідно на випадок збою під час зміни структури колонок.
2.  **Перевірка сумісності**:
    - Перевірте файл `package.json` на наявність оновлень критичних залежностей.
    - Переконайтеся, що версія Node.js на build-машині відповідає вимогам проєкту.
3.  **Планування часу простою**:
    - Оскільки GitHub Pages використовує атомарне розгортання, час простою (downtime) зазвичай становить 0 секунд. Проте при великих міграціях даних у Google Sheets рекомендується проводити оновлення у години найменшої активності користувачів.

## 2. Процес оновлення
Для розгортання нової версії виконайте наступні кроки:

1.  **Отримання нового коду**:
    ```bash
    git pull origin main
    ```
2.  **Зупинка служб**:
    - Для даного типу проєкту зупинка сервера не потрібна. 
3.  **Оновлення залежностей та збірка**:
    ```bash
    npm install
    npm run build
    ```
4.  **Міграція даних та конфігурацій**:
    - Якщо нова версія коду вимагає нових колонок у Google Sheets, внесіть зміни до структури таблиці **до** моменту деплою фронтенду.
    - Оновіть URL Google Apps Script у файлі конфігурації, якщо скрипт був переопублікований як нова версія.
5.  **Розгортання нового коду**:
    ```bash
    npm run deploy
    ```

## 3. Перевірка після оновлення
1.  Очистіть кеш браузера або відкрийте сайт у режимі інкогніто.
2.  Перевірте коректність відображення нових функцій.
3.  Перевірте консоль розробника на відсутність помилок запитів до API (403, 404, 500).

## 4. Процедура відкату (Rollback) 🔙
Якщо після оновлення виявлено критичні помилки, які неможливо виправити оперативно, виконайте процедуру повернення до стабільної версії:

1.  **Пошук стабільної версії**: Визначте останній робочий тег або хеш коміту:
    ```bash
    git tag
    ```
2.  **Перемикання на стабільний код**:
    ```bash
    git checkout v1.0.0  # замініть на назву вашого тегу
    ```
3.  **Перескладання та деплой**:
    ```bash
    npm install
    npm run build
    npm run deploy
    ```
4.  **Відновлення даних**: Якщо структура таблиці була пошкоджена, відновіть дані з резервної копії Google Sheets, створеної на етапі підготовки.
