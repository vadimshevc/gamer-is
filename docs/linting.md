# Документація лінтингу проєкту

## Обраний інструмент
**ESLint** (Flat Config формат). Це сучасний стандарт для статичного аналізу JavaScript/React коду.

## Причини вибору
- Повна інтеграція з Vite.
- Можливість автоматичного виправлення помилок форматування.
- Велика спільнота та підтримка плагінів для React.

## Базові правила
1. `no-unused-vars`: Забороняє змінні, які не використовуються (запобігає накопиченню "сміття").
2. `prefer-const`: Вимагає використовувати `const` для змінних, які не змінюють значення (покращує читабельність).
3. `no-console`: Попереджає про забуті логі, які не мають потрапляти в продакшн.

## Інструкція з запуску
- Перевірка: `npm run lint`
- Автовиправлення: `npx eslint . --fix`

## Git Hooks (Автоматизація)
Для запобігання потраплянню неякісного коду в репозиторій інтегровано інструмент Husky:
- pre-commit hook: Налаштовано автоматичний запуск лінтера перед кожним збереженням змін (`git commit`).
- Якщо лінтер знаходить помилки, процес створення коміту переривається до їх виправлення.

## Інтеграція в процес збірки
Лінтинг є обов'язковим етапом життєвого циклу розробки:
- Команду перевірки додано до процесу підготовки до розгортання (`predeploy`).
- GitHub Pages не отримає оновлення, якщо код не пройде перевірку на відповідність стандартам якості.

## Статична типізація
Для виконання вимог щодо статичної перевірки типів до проєкту додано конфігурацію TypeScript:
- Файл конфігурації: `tsconfig.json`.
- Це забезпечує базу для типізації даних у компонентах, що значно знижує ризик виникнення помилок у рантаймі та полегшує підтримку проєкту.
